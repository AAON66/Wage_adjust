## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend file-list payload -> frontend shared types | sharing badge / cleanup notice contract 若被错误建模，会导致错误文件显示状态或提示 |
| shared `FileList` -> `MyReview` + `EvaluationDetail` | 一个组件同时服务 requester 和管理员页面，任何 badge 逻辑都会被两个页面共同消费 |
| requester cleanup notices -> MyReview toast surface | 页面需要解释“文件为何消失”，但不能因为重复渲染或错页渲染制造误导 |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-05 | Information Disclosure | `frontend/src/components/evaluation/FileList.tsx` | mitigate | 仅根据后端返回的 per-file `sharing_status === 'pending'` 渲染 `待同意`，不要从 submission 级别状态自行推断，避免标签泄漏到原上传者文件或无关文件 |
| T-21-06 | Integrity | `frontend/src/pages/MyReview.tsx` cleanup notice wiring | mitigate | cleanup 原因只消费当前 submission 的 response-level notices，并对重复提示做去重，避免 stale timeout 状态反复提示或提示到错误页面 |

| T-21-08 | Tampering / Policy Bypass | `frontend/src/services/sharingService.ts` + sharing history UI | mitigate | 删除 `revokeSharingRejection` 调用和 `撤销拒绝` CTA，避免前端继续暴露与 D-07 冲突的 rejected->pending 恢复路径 |
