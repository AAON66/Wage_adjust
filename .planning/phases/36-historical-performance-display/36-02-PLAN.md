<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| 前端 fetch → FastAPI router | path 参数 employee_id 由前端提供，可伪造任意 UUID |
| require_roles → AccessScopeService | 角色通过后仍需部门级过滤 |
| AccessScopeService → Employee 表查询 | 员工不存在 vs 无权限 不能泄露差异（现状：两者都不暴露存在性） |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-36-02-01 | Elevation of Privilege | manager 构造跨部门 employee_id | mitigate | `AccessScopeService.ensure_employee_access(current_user, employee_id)` 基于 `current_user.departments` 做 department name set 匹配；跨部门抛 PermissionError → 403（D-06）；Task 3 覆盖用例 `_manager_cross_department_returns_403` |
| T-36-02-02 | Spoofing | employee 直访 /records/by-employee/{id} 读自己数据 | mitigate | `require_roles('admin','hrbp','manager')` 白名单兜底；**不依赖** AccessScopeService 的 `current_user.employee_id == employee.id` 放行分支（该分支仅适用于 `/me/*` 无参端点）；Task 3 覆盖用例 `_employee_role_returns_403` |
| T-36-02-03 | IDOR / Information Disclosure | employee_id 是 UUID 可枚举 | mitigate | AccessScopeService 返回 None（员工不存在）时 router 统一转 404；PermissionError 时统一转 403；两者 detail 文本不同但 status code 区分 — 权衡：本期保持与 evaluations 端点同款（404 vs 403 可区分），不做「统一 404 掩盖存在性」再加工，避免偏离 codebase 惯例（参考 evaluations.py line 104-119） |
| T-36-02-04 | Information Disclosure | 敏感评语（comment）通过 API 返回给跨部门 manager | mitigate | AccessScopeService 跨部门直接 403，manager 永远看不到他部门员工的 comment 字段 |
