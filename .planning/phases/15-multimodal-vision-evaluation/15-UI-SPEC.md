# Phase 15 — UI Design Contract

> Visual and interaction contract for the multimodal vision evaluation phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope Note

Phase 15 is **backend-dominant**. The four VISION requirements (01-04) concern PPT image extraction, vision API integration, structured JSON output, and batch failure isolation -- all implemented in Python parsers, LLM service, and evaluation engine.

**Frontend surface area is minimal:** Vision evaluation results flow into the existing `EvidenceCard` component as new `EvidenceItem` records with `source_type: 'vision_evaluation'` or `'vision_failed'`. The existing card already renders dynamic metadata, confidence bars, tags, and expand/collapse. This contract defines only the incremental visual treatment needed for vision-sourced evidence.

## Interaction Contract

### Vision Evidence in EvidenceCard

1. **No new interactions required.** Vision evidence appears in the same `EvidenceCard` expand/collapse pattern as all other evidence types.
2. **Confidence bar** reflects `confidence_score` which is `quality_score / 5.0` (normalized 0-1). The existing green/yellow/red thresholds (>=75%, >=50%, <50%) apply directly.
3. **Tags section** displays dimension relevance as chips when `vision_dimension_relevance` is present in metadata.
4. **Integrity flag** (`prompt_manipulation_detected`) applies to vision evidence the same as text evidence -- red border and review notice.

### Batch Processing Visibility

Vision evaluation happens server-side during parsing. The frontend shows results only after parsing completes. The existing `parsing` -> `parsed`/`failed` status flow in `StatusIndicator` covers this.

- A submission with mixed success/failure: individual `EvidenceItem` records appear per image. Failed images have `source_type: 'vision_failed'`, `confidence_score: 0`, and `vision_failed: true` in metadata.
- The user sees both successful vision evidence cards and failed ones in the same evidence list.
