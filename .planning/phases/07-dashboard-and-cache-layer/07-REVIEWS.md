# 跨 AI 计划评审 — Phase 7

> 注：Claude CLI 和 Gemini CLI 在当前环境不可用。仅使用 Codex (GPT-5.4) 完成评审。

## Codex 评审 (GPT-5.4)

### Plan 01: 后端

**概要**
方向正确：SQL 聚合、Redis 缓存、专用 API 端点。主要问题是缓存和端点策略不够精确，无法保证正确的权限范围和数据新鲜度。

**优势**
- 将聚合推到 SQL 侧（DASH-01 正确修复）
- Redis 不可用时返回 503，符合 D-03
- 将看板数据拆分为专用 API 而非单一响应
- 包含测试

**关注点**
- `[HIGH]` 缓存 key 仅含 cycle_id + role 不足——同角色不同 manager 可能有不同数据范围，导致数据泄漏
- `[HIGH]` 待审批数 30 秒刷新与 Redis TTL 5-15 分钟直接冲突——需单独的新鲜度策略（不缓存或极短 TTL）
- `[HIGH]` D-11 员工不可见看板必须在后端 API 层强制执行（403），不能仅靠前端隐藏
- `[MEDIUM]` 无缓存失效策略——审批操作、评估更新、导入、周期变更都会导致缓存过期
- `[MEDIUM]` 部门下钻需要参数化端点（按 department_id）
- `[MEDIUM]` SQL 聚合如果缺少相关索引仍可能慢

**风险评估：HIGH**

### Plan 02: 前端组件

**概要**
范围清晰，组件化合理。主要缺口在 UX 错误状态、轮询与周期切换的异步协调、前端行为与后端新鲜度约束的匹配。

**优势**
- 统一使用 ECharts
- 组件与 DASH-03 ~ DASH-06 清晰映射
- 可复用的 usePolling hook
- 独立的 dashboard API 客户端

**关注点**
- `[HIGH]` 未处理 503 Redis 不可用的前端错误状态（D-03 使其成为常见故障模式）
- `[MEDIUM]` 轮询与周期切换/页面卸载的竞态条件
- `[MEDIUM]` 部门下钻的状态模型未描述（展开/收起、周期切换重置）
- `[MEDIUM]` KPI 卡片同时包含实时和缓存指标，统一获取可能导致不必要的重载

**风险评估：MEDIUM**

## 共识总结

### 核心关注点（按优先级）

1. **[HIGH] 缓存 key 范围不足** — `cycle_id + role` 对 manager 角色不够（不同 manager 管不同部门），可能导致跨部门数据泄漏。应扩展为 `cycle_id + user_id` 或 `cycle_id + scope_id + role`
2. **[HIGH] 待审批数新鲜度冲突** — 30 秒轮询 vs 5-15 分钟 TTL 直接矛盾。待审批数应不缓存或使用极短 TTL
3. **[HIGH] 员工角色排除需后端强制** — D-11 必须在 API 层返回 403，不能仅靠前端路由隐藏
4. **[HIGH] 前端未处理 503 错误状态** — Redis 不可用是预期故障模式，需显式 UI 处理
5. **[MEDIUM] 缓存失效策略缺失** — 审批、评估、导入操作后缓存未清除
6. **[MEDIUM] 周期切换/轮询竞态** — 需请求取消和状态重置

### 已确认的优势

- 三计划 Wave 分层（后端/组件/集成）结构清晰
- SQL 聚合替换全表扫描方向正确
- Redis 503 强制策略符合 D-03
- ECharts 统一采用合理
- usePolling hook 可复用设计

### 分歧观点

无（仅有一个评审者）。
