1. 创建 `frontend/Dockerfile`（per D-06 多阶段构建）：

```dockerfile
# Stage 1: 构建前端
FROM node:18-alpine AS build

WORKDIR /app

COPY package.json package-lock.json ./
RUN npm ci

COPY . .

# 构建时 API 地址使用相对路径，由 Nginx 反向代理到 backend
ARG VITE_API_BASE_URL=/api/v1
ENV VITE_API_BASE_URL=$VITE_API_BASE_URL

RUN npm run build

# Stage 2: Nginx 托管
FROM nginx:alpine

# 复制构建产物到 Nginx 默认目录
COPY --from=build /app/dist /usr/share/nginx/html

# 复制自定义 Nginx 配置
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 8080

CMD ["nginx", "-g", "daemon off;"]
```

2. 创建 `frontend/nginx.conf`（per D-07 反向代理 + 静态文件，复用 deploy/deploy.sh 的 Nginx 配置思路）：

```nginx
server {
    listen 8080;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # SPA 路由 fallback
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 反向代理（per D-07）
    location /api/ {
        proxy_pass http://backend:8011;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 文件上传大小限制
        client_max_body_size 200M;

        # AI 调用超时较长
        proxy_read_timeout 300s;
        proxy_connect_timeout 10s;
        proxy_send_timeout 300s;
    }

    # 健康检查端点代理
    location /health {
        proxy_pass http://backend:8011;
        proxy_set_header Host $host;
    }

    # 静态文件缓存（复用 deploy.sh 策略）
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # gzip 压缩
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml text/javascript image/svg+xml;
    gzip_min_length 256;
}
```

关键设计点：
- `VITE_API_BASE_URL=/api/v1` 使用相对路径，前端请求 `/api/v1/...` 由 Nginx proxy_pass 到 `backend:8011`
- Docker 内部网络中 backend 是 docker-compose 定义的服务名，自动解析
- 超时 300s 与 deploy.sh 一致，适配 AI 长时间调用
- client_max_body_size 200M 与 config.py 的 max_upload_size_mb=200 一致

1. 创建 `docker-compose.prod.yml`（per D-01, DEPLOY-04）— 生产环境专用：

```yaml
services:
  redis:
    image: redis:7-alpine
    restart: always
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 3

  backend:
    build:
      context: .
      dockerfile: Dockerfile
    restart: always
    env_file:
      - .env
    environment:
      REDIS_URL: redis://redis:6379/0
      ENVIRONMENT: production
    depends_on:
      redis:
        condition: service_healthy
    volumes:
      - uploads_data:/app/uploads
    healthcheck:
      test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:8011/health')"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 30s

  celery-worker:
    build:
      context: .
      dockerfile: Dockerfile
    command: celery -A backend.app.celery_app worker --loglevel=info --concurrency=2
    restart: always
    env_file:
      - .env
    environment:
      REDIS_URL: redis://redis:6379/0
      ENVIRONMENT: production
    depends_on:
      redis:
        condition: service_healthy
      backend:
        condition: service_healthy

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
      args:
        VITE_API_BASE_URL: /api/v1
    restart: always
    ports:
      - "8080:8080"
    depends_on:
      backend:
        condition: service_healthy

volumes:
  redis_data:
  uploads_data:
```

关键设计点：
- 不含 PostgreSQL 容器（per D-03，使用外部 PG，通过 .env 中 DATABASE_URL 连接）

- frontend 唯一对外暴露 8080 端口（per D-08）
- uploads 使用 named volume 持久化（不用 bind mount 避免权限问题）
- backend 有健康检查，celery-worker 和 frontend 依赖 backend healthy
- 所有服务 restart: always 确保自动恢复
- celery-worker 复用 backend 镜像，只覆盖 command

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| internet → Nginx | 用户请求通过 8080 端口进入 Nginx 容器 |
| Nginx → backend | Nginx 通过 Docker 内部网络代理到 backend:8011 |
| backend → external PG | 后端通过 DATABASE_URL 连接外部 PostgreSQL |
| backend → Redis | 后端和 celery-worker 通过内部网络连接 Redis |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-04 | I (Information Disclosure) | Nginx | mitigate | 仅暴露 8080 端口，backend 端口不对外；nginx.conf 不含 server_tokens |
| T-24-05 | D (Denial of Service) | Nginx | mitigate | client_max_body_size 200M 限制上传大小；proxy_read_timeout 300s 防止无限等待 |
| T-24-06 | T (Tampering) | Docker images | accept | 使用官方 python:3.9-slim 和 nginx:alpine 基础镜像 |
| T-24-07 | I (Information Disclosure) | .env | mitigate | .dockerignore 排除 .env，docker-compose 通过 env_file 引用而非 COPY 进镜像 |
| T-24-08 | E (Elevation of Privilege) | containers | mitigate | gunicorn worker 以非 root 运行（默认行为）；Nginx alpine 镜像使用 nginx 用户 |
</threat_model>
