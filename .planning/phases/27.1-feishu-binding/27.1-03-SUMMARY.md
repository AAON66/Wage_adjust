# Phase 27.1 Plan 03: 设置页飞书账号绑定/解绑 Summary

Front-end integration of Plan 02's REST contract: Settings page gets a new "飞书账号" section with bind/unbind entry, a new `/settings/feishu-bind-callback` callback page completes the OAuth roundtrip, and `feishuErrors.ts` gains two new categorized error codes. Tasks 1-3 are code-complete; Task 4 is a `checkpoint:human-verify` gate (8-step Manual UAT) that must be driven by the user.

---

## What Was Built

### Task 1: `services/auth.ts` + `utils/feishuErrors.ts` (commit `acc293c`)

- **`authorizeFeishu(purpose?: 'login' | 'bind')`** — added optional keyword argument. `undefined` omits the query param (backend defaults to `login` per Plan 02), so Phase 27 callers that pass nothing stay byte-compatible.
- **`bindFeishu(code, state): Promise<UserProfile>`** — wraps `POST /auth/feishu/bind` and returns the updated `UserProfile` (with fresh `feishu_open_id`).
- **`unbindFeishu(): Promise<UserProfile>`** — wraps `POST /auth/feishu/unbind`; returns the user with `feishu_open_id: null`.
- **`FeishuErrorCode`** — adds `'feishu_already_bound'` and `'feishu_employee_mismatch'` in the ordered union.
- **`COPY` map** — adds the two new codes with the copy verbatim matching backend detail literals (required for keyword matching in classifier):
  - `feishu_already_bound: '该飞书账号已被其他系统账号绑定，请先在原账号解绑'`
  - `feishu_employee_mismatch: '飞书工号与当前账号绑定的工号不一致，无法绑定'`
- **`classifyBackend`** — inserts `if (status === 409) return 'feishu_already_bound'` and extends the `status === 400` branch with:
  1. `detail.includes('工号与当前账号')` → `'feishu_employee_mismatch'` (D-06)
  2. `detail.includes('当前账号未关联员工信息')` → `'feishu_employee_mismatch'` (D-07 — consolidated)
  3. existing `'state'` / `'工号'` / `'授权码'` keywords preserved
  Order is critical: specific bind-path substrings win over the generic `'工号'` keyword, keeping Phase 27 login path `'工号未匹配，请联系管理员开通'` mapped to `employee_not_matched` unchanged.

### Task 2: `pages/FeishuBindCallbackPage.tsx` + `App.tsx` (commit `2b6ab54`)

- **New page** modeled after `FeishuCallbackPage.tsx`:
  - Three-state machine (`processing` / `success` / `failed`)
  - Reads `code` + `state` from `useSearchParams`
  - Calls `bindFeishu(code, state)` → `refreshProfile()` → `navigate('/settings', { replace: true, state: { bindSuccess: true } })`
  - On failure: renders `resolveFeishuError('backend', err).message` (categorized copy, React-escaped) + `返回设置` button
  - `hasRunRef` guard plus `replace: true` (T-27.1-11: prevents double-fire under StrictMode and blocks code/state history replay)
  - Uses `eyebrow: 飞书绑定` / `正在完成飞书绑定…` / `绑定成功，正在返回设置页…` / `无法完成绑定` copy
- **`App.tsx`** — import added in alphabetical order; new `Route` registered inside `ProtectedRoute allowedRoles={["admin","hrbp","manager","employee"]}` block (same block as `/settings`). This ensures T-27.1-10 (unauthenticated access redirected to `/login` by `ProtectedRoute`).

### Task 3: `pages/Settings.tsx` FeishuBindingSection (commit `97e54e1`)

- **Imports** extended: `authorizeFeishu` + `unbindFeishu` from `services/auth`; `resolveFeishuError` from `utils/feishuErrors`; `useEffect` added.
- **New state**: `feishuError`, `feishuSuccess`, `isFeishuProcessing`.
- **New `useEffect`** — on mount / `location.state` change, reads `location.state.bindSuccess`; if truthy, sets `'飞书账号已绑定'` banner and clears history state via `window.history.replaceState({}, '')` so a page refresh doesn't resurface the banner.
- **Two new handlers**:
  - `handleFeishuBind()` → `authorizeFeishu('bind')` → `window.location.href = authorize_url` (full-page redirect; state is preserved server-side via Redis). Any network error before the redirect maps to a categorized inline banner.
  - `handleFeishuUnbind()` → native `window.confirm('解绑后你将无法再使用飞书账号登录，需重新绑定。确认解绑？')` (D-10) → `unbindFeishu()` → `refreshProfile()` → `'已解除飞书绑定'` success banner.
- **New third top-level `<section>`** placed after the password management section (D-11):
  - `eyebrow: 登录方式`, `h2: 飞书账号`
  - Copy varies with `user.feishu_open_id`: bound vs not-bound
  - Pill-shaped status badge: green `已绑定` / grey `未绑定`
  - Single action button: primary `使用飞书绑定` or secondary `解除飞书绑定` based on binding state
  - Inline error/success banners share the exact style (success-bg / danger-bg) used by the existing password-change section
- **Explicit security check**: UI renders ONLY the badge text (`已绑定` / `未绑定`); `user.feishu_open_id` is NEVER projected into the DOM (T-27.1-05).

## Threat Mitigation Landing Points (T-27.1 register, front-end side)

| Threat ID | Category | Landing Point |
|-----------|----------|---------------|
| T-27.1-01 | 跨用户绑定 (Elevation of Privilege) | Settings.tsx has no bypass path; the Bind button only calls `authorizeFeishu('bind')` — backend Plan 02 enforces `employee_no` match |
| T-27.1-02 | open_id 盗用 | FeishuBindCallbackPage surfaces 409 as `feishu_already_bound` copy with no force-bind UI |
| T-27.1-03 | CSRF on unbind | Accept — POST + JWT Bearer header (Axios adds it) is not a simple request; `window.confirm` is UX confirmation, not a security layer |
| T-27.1-04 | state 混用 | Front end only passes `code` + `state` to backend; state-prefix isolation is enforced in Plan 01 service layer |
| T-27.1-05 | open_id 泄漏 | Settings.tsx renders ONLY status badge text; `user.feishu_open_id` never flows into DOM |
| T-27.1-06 | XSS 错误文案注入 | All error copy flows through `resolveFeishuError('backend', err).message` (static COPY map); React escapes via text nodes; `grep dangerouslySetInnerHTML\|innerHTML` → 0 matches in new files |
| T-27.1-10 | 路由权限缺失 | `/settings/feishu-bind-callback` registered under `ProtectedRoute allowedRoles=["admin","hrbp","manager","employee"]` — unauthenticated access redirects to `/login` (Step 7 of Task 4 UAT) |
| T-27.1-11 | history stack 污染 (code/state 回退重放) | FeishuBindCallbackPage uses `navigate(..., { replace: true })` on both success and failure; the `?code=...&state=...` URL never enters the back stack |
