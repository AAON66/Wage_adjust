## 后端端点设计 & 回调分流

### Q1: 后端绑定端点如何设计？

| Option | Description | Selected |
|--------|-------------|----------|
| 新增独立绑定端点 (Recommended) | POST /feishu/bind + POST /feishu/unbind（需 JWT）。/authorize 共用不分 intent | ✓ |
| 复用 /feishu/callback 按 JWT 分流 | 同一端点内部检查 JWT；权限富含，难审计 | |

**User's choice:** 新增独立绑定端点

### Q2: 前端回调路由怎么分？

| Option | Description | Selected |
|--------|-------------|----------|
| 新增 /settings/feishu-bind-callback 路由 (Recommended) | 独立 FeishuBindCallbackPage；与登录回调完全解耦 | ✓ |
| 复用 /auth/feishu/callback 根据登录态分流 | 组件内部分叉；逻辑臃肿 | |
| 用同一路由 + query 参数区分 | intent=bind 参数；排错复杂 | |

**User's choice:** 新增 /settings/feishu-bind-callback 路由

### Q3: 如果待绑定的 feishu_open_id 已被其他系统账号占用？

| Option | Description | Selected |
|--------|-------------|----------|
| 返回 409 + 中文错误 (Recommended) | HTTP 409 + 「该飞书账号已被其他系统账号绑定」；feishu_already_bound 新 code | ✓ |
| 强制覆盖 | 原账号清空 feishu_open_id；有恶意剥夺风险 | |
| 返回表单让用户选择 | 转移绑定流程；UX 不友好 | |

**User's choice:** 返回 409 + 中文错误

## 绑定安全语义

### Q1: 手动绑定时是否校验 employee_no 与当前用户绑定的 Employee 匹配？

| Option | Description | Selected |
|--------|-------------|----------|
| 必须匹配，不匹配拒绝 (Recommended) | 校验 leading-zero 容差后必须等于 User.employee.employee_no；严防绑他人 | ✓ |
| 不校验，仅依赖 open_id 唯一约束 | 任何人能绑任何飞书账号；严重安全漏洞 | |
| 管理员可关闭校验 | 扩展点；不在 Phase 27.1 范围 | |

**User's choice:** 必须匹配，不匹配拒绝

### Q2: 已绑定用户再点「绑定」按钮时的行为？

| Option | Description | Selected |
|--------|-------------|----------|
| 按钮改「解绑」禁止直接更换 (Recommended) | 先解绑再重绑；符合「一个账号一个飞书」模型 | ✓ |
| 按钮改「更换」，弹窗确认后原子更换 | 一步到位；审计难分辨更换 vs 新绑 | |
| 按钮 disabled，只能解绑 | 更明显但交互多一步 | |

**User's choice:** 按钮改「解绑」禁止直接更换

## 解绑流程 & 设置页 UI

### Q1: 解绑操作要多安全？

| Option | Description | Selected |
|--------|-------------|----------|
| 弹窗二次确认 + 保留 session (Recommended) | 确认对话框 → 清空 feishu_open_id → toast。session 不注销 | ✓ |
| 弹窗二次确认 + 强制注销 session | token_version+1 全局踢出；对用户不友好 | |
| 仅 inline 确认（不弹窗） | 两次点击确认；误操作风险高 | |

**User's choice:** 弹窗二次确认 + 保留当前 session

### Q2: 设置页「飞书绑定」section 的位置与形态？

| Option | Description | Selected |
|--------|-------------|----------|
| 密码修改下方新增独立 section (Recommended) | 节奏自然，视觉对等 | ✓ |
| 新建「登录方式」卡片统合 | 语义清晰但重构较大 | |
| 单独放在页面最上方 | 过度引导 | |

**User's choice:** 密码修改下方新增独立 section

### Q3: 当前绑定状态如何展示？

| Option | Description | Selected |
|--------|-------------|----------|
| 仅状态标签 + 提示性文案 (Recommended) | 绿/灰 badge + 中文说明 + 动态按钮；不暴露 open_id | ✓ |
