<decisions>
## Implementation Decisions

### 后端端点设计 & 回调分流
- **D-01:** 新增两个独立后端端点：
  - `POST /api/v1/auth/feishu/bind` — 需 JWT，body `{code, state}`，调用飞书换 token/user_info 后将 `feishu_open_id` 写入 `request.user.id`。
  - `POST /api/v1/auth/feishu/unbind` — 需 JWT，无 body，将 `request.user.feishu_open_id` 设为 NULL。
  现有 `/feishu/callback` 保持公开端点不变（仅用于登录）。
- **D-02:** `GET /api/v1/auth/feishu/authorize` **扩展**为接受 `?purpose=login|bind` 查询参数（默认 `login` 向后兼容）：
  - `purpose=login`：返回的 authorize_url 的 `redirect_uri` = `FEISHU_REDIRECT_URI`（指向 `/auth/feishu/callback`）；state 存到 Redis key `feishu_oauth_state:{state}`。
  - `purpose=bind`：返回的 authorize_url 的 `redirect_uri` = `FEISHU_BIND_REDIRECT_URI`（新增 Settings，指向 `/settings/feishu-bind-callback`）；state 存到 Redis key `feishu_oauth_state:bind:{state}`。
- **D-03:** 前端新增独立路由 `/settings/feishu-bind-callback` 对应新组件 `FeishuBindCallbackPage`。三态 UI（处理中 / 成功 / 失败），成功后 `navigate('/settings', { replace: true, state: { bindSuccess: true } })`，`SettingsPage` 监听 state 显示 toast。
- **D-04:** 需在飞书开放平台后台登记**第二个** Redirect URL：`http://localhost:5174/settings/feishu-bind-callback`（dev）+ 对应生产域名值。发版审批流程与登录回调相同。
- **D-05:** 冲突处理：`/feishu/bind` 检测到 `feishu_open_id` 已被其他 User 占用 → 返回 **HTTP 409 Conflict** + `{error: 'http_error', message: '该飞书账号已被其他系统账号绑定，请先在原账号解绑'}`。前端 `feishuErrors.ts` 新增 code `feishu_already_bound`。

### 绑定安全语义
- **D-06:** 手动绑定**必须**校验飞书账号的 `employee_no` 与当前 User 已绑定 `Employee.employee_no` 一致（复用 Phase 26 的 leading-zero 容差匹配）。不一致 → 返回 HTTP 400 + 「飞书工号与当前账号绑定的工号不一致，无法绑定」。防止用户 A 把同事 B 的飞书账号绑到自己的系统账号下。
- **D-07:** 当前 User 未关联 Employee 记录（`user.employee_id IS NULL`）时，不允许绑定飞书 → 返回 HTTP 400 + 「当前账号未关联员工信息，请联系管理员」。
- **D-08:** 已绑定用户再点「绑定」按钮——设置页直接显示「**解除飞书绑定**」按钮，不提供「更换」路径。想换账号的流程：先解绑 → 再重新绑定新的飞书账号。
- **D-09:** 新增 code `feishu_employee_mismatch` 到 `feishuErrors.ts`（400），对应 D-06 的文案。原有 `employee_not_matched` 保留用于**登录**场景（后端未找到任何 employee_no 对应的 User，Phase 27 已实现）。

### 解绑流程 & 设置页 UI
- **D-10:** 解绑流程：点击「解除飞书绑定」按钮 → 弹**确认对话框**（浏览器原生 `window.confirm` 或简单自定义 Modal，与项目现有 UX 节奏一致）→ 文案「解绑后你将无法再使用飞书账号登录，需重新绑定。确认解绑？」→ 确认后 `POST /feishu/unbind` → 清空 `feishu_open_id` → 显示成功 toast「已解除飞书绑定」。**保留当前 session**（不触碰 token_version），用户继续工作不被踢出。
- **D-11:** `SettingsPage` 结构：出生/手机号等个人信息 → 密码修改 → **飞书绑定（新增独立 section）**。section 使用 `surface` 外框 + `eyebrow: '登录方式'` + 标题「飞书账号」+ 状态说明 + 按钮。视觉与密码修改 section 对等。
- **D-12:** 状态展示：仅状态标签 + 提示性文案（不暴露 `open_id`、不查飞书获取真实名/邮箱）：
  - **已绑定**：绿色 badge「已绑定」+ 文案「你的飞书账号已绑定到系统账号，可在登录页使用飞书扫码登录」+ 按钮「解除飞书绑定」
  - **未绑定**：灰色徽标「未绑定」+ 文案「绑定飞书账号后，可用飞书扫码快速登录」+ 按钮「使用飞书绑定」
- **D-13:** 绑定按钮点击 → 调 `GET /api/v1/auth/feishu/authorize?purpose=bind` 拿 `authorize_url` → `window.location.href = authorize_url` 整页跳转飞书授权页。与 Phase 27 登录按钮的交互模式完全一致。

### 入口可见性 & 审计
- **D-14:** 所有登录角色（admin/hrbp/manager/employee）都可见飞书绑定 section。不引入额外的权限分层：任何能登录到系统的用户都可以为自己的账号绑定飞书。
- **D-15:** 绑定/解绑写入 AuditLog：
  - 新增 action enum `feishu_bound` 和 `feishu_unbound`（若 AuditLog 当前 enum 不支持，需扩展模型）
  - 记录字段：`user_id`、`action`、`timestamp`、`ip_address`、`details = {open_id_prefix: <前8位>, open_id_suffix: <后8位>}`（不存完整 open_id，保护隐私）

### Claude's Discretion
- Redis state 生命周期策略细节（绑定与登录的 TTL 是否相同——建议相同 300s 保持一致）
- `FeishuBindCallbackPage` 的骨架屏与错误卡片精确视觉（沿用 `FeishuCallbackPage` 模板）
- 确认对话框实现（浏览器原生 `confirm` vs 自定义 Modal，取决于项目是否已有 Modal 组件——若没有就用原生）
- 后端 `_bind_open_id_to_user` 私有方法的具体实现细节与事务处理
- 单元测试 mock 策略（复用 Phase 26 测试模板）
- Success toast 实现（可用现有 banner 或最小自造，不引入全局 Toast 组件库）

<specifics>
## Specific Ideas

- 绑定/解绑完全是登录后的动作，**不**走公开端点（/feishu/callback 保持只用于登录）
- 两套 state key 前缀（`feishu_oauth_state:{state}` vs `feishu_oauth_state:bind:{state}`）天然隔离，绑定 state 不能被登录端点误消费反之亦然
- 绑定成功后 `refreshProfile()` 即可让 SettingsPage 的状态从「未绑定」切到「已绑定」，不需要刷新整页
- 解绑不动 `token_version`：保留用户当前会话，体验更平滑
- 文案要求：绑定/解绑任何错误都走 `feishuErrors.ts`，UI 层永远拿已经是中文的 `{message}` 字段

</specifics>
