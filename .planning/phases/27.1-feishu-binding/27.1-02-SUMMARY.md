# Phase 27.1 Plan 02: feishu/bind + feishu/unbind API 端点与审计日志 Summary

在 `backend/app/api/v1/auth.py` 新增两个已登录端点并扩展 authorize 端点的 `purpose` 查询参数，把 Plan 01 的 service 方法通过 FastAPI 路由暴露为 HTTP 接口；审计日志在 handler 层写入，open_id 按 D-15 截断到前 8 + 后 8。

---

## What Was Built

### Task 1: auth.py 端点扩展与新增（commit `1be9ea1`）

- **Import 扩展** — 在 `from backend.app.models.user import User` 后新增 `from backend.app.models.audit_log import AuditLog`
- **GET /feishu/authorize 扩展**：新增 `purpose: str = Query('login', pattern='^(login|bind)$')`
  - 默认 `'login'`，保持 Phase 27 前端（不传 purpose）的字节级兼容
  - 非白名单值由 FastAPI Query pattern 拦截为 422
  - 将 `purpose` 透传给 `service.generate_authorize_url(redis_client, purpose=purpose)`
- **POST /feishu/bind**（新增）
  - `Depends(get_current_user)` → 无 JWT 自动 401（oauth2_scheme 行为）
  - body `FeishuCallbackRequest{code, state}`（复用 Phase 26 schema）
  - 调 `service.handle_bind_callback(code, state, current_user, redis_client)`
  - handler 层写 `AuditLog(action='feishu_bound', target_type='user', target_id=user.id, detail={open_id_prefix, open_id_suffix, ip_address}, operator_id, operator_role)`
  - `db.commit(); db.refresh(user)` → 返回 `UserRead.model_validate(user)`（含最新 `feishu_open_id`）
- **POST /feishu/unbind**（新增）
  - `Depends(get_current_user)` → 无 JWT 自动 401
  - 无 body
  - 先读取 `current_user.feishu_open_id` 作为 `previous_open_id`（用于审计截断），再调 `service.handle_unbind(current_user)` 清空
  - 写 `AuditLog(action='feishu_unbound', ...)`，detail 用 previous_open_id 截断值
  - `db.commit(); db.refresh(user)` → 返回 `UserRead.model_validate(user)`（含 `feishu_open_id=None`）
  - **不触碰 `token_version`**（D-10：解绑不踢会话）
- **`_truncate_open_id(open_id)` helper**（新增）
  - 空串 / None → `{'open_id_prefix': '', 'open_id_suffix': ''}`
  - `len <= 16` 分支：前后各半（飞书真实 open_id 长度 >= 28，此分支仅作安全兜底）
  - `len > 16` 分支：`{'open_id_prefix': open_id[:8], 'open_id_suffix': open_id[-8:]}`
- **既有 `/feishu/callback` 端点未被改动**（只扩展 authorize + 新增 bind/unbind）

## Plan 03 依赖的 REST 契约

Plan 03（前端）应按下列契约调用：

```
GET  /api/v1/auth/feishu/authorize?purpose=bind
     → 200 { authorize_url: string, state: string }
     → 422 { error: 'validation_error', ... }（非法 purpose）
     → 503 { error: 'http_error', message: '绑定服务暂不可用...' }（未配置 FEISHU_BIND_REDIRECT_URI）

POST /api/v1/auth/feishu/bind
     Headers: Authorization: Bearer <jwt>
     Body:    { code: string, state: string }
     → 200 UserRead（含更新后的 feishu_open_id）
     → 401 Could not validate credentials（无 JWT / token 失效）
     → 400 '无效的 state 参数...' | '当前账号未关联员工信息...' | '飞书工号与当前账号绑定的工号不一致...' | '授权码已使用'
     → 409 '该飞书账号已被其他系统账号绑定，请先在原账号解绑'
     → 502 飞书服务异常
     → 503 Redis / 绑定服务不可用

POST /api/v1/auth/feishu/unbind
     Headers: Authorization: Bearer <jwt>
     Body:    （无）
     → 200 UserRead（feishu_open_id: null）
     → 401 Could not validate credentials
```

Plan 03 前端 `feishuErrors.ts` 需新增映射：
- `feishu_already_bound` (409 + `'该飞书账号已被其他系统账号绑定'`)
- `feishu_employee_mismatch` (400 + `'飞书工号与当前账号绑定的工号不一致'`)

Plan 03 应在 `useAuth` 扩展 `bindFeishu(code, state)` / `unbindFeishu()` / `refreshProfile()` 方法；绑定成功后 `refreshProfile()` 让 SettingsPage 的「已绑定」状态即时生效。

## Deviations from Plan

**无主要偏离。**

两处微调（Rule 2 加固）：

1. **`feishu_unbind` 接受 `settings` 依赖参数** — Plan 原文建议 unbind 不接 settings（「无需 Redis，纯 DB 操作」）。实际我保留 `settings=Depends(get_app_settings)` 传给 `FeishuOAuthService(db, settings=settings)`，理由：
   - 保持 service 构造签名一致，未来若 `handle_unbind` 新增 settings 依赖（如写入飞书端的解绑通知），无需改 handler
   - 对当前请求开销可忽略（get_app_settings 是 LRU 缓存）
   - 不影响任何验收条件或行为
   - 记录于 `decisions` 字段
2. **Task 2 未单独提交 RED commit** — Plan 原文 `tdd="true"`，但 Plan 02 Task 1 已完成实现，意味着 "RED" 阶段的测试不会失败（测试当作端到端回归验证）。合并为单个 `test(27.1-02): ...` GREEN commit，注释明确测试用途。
