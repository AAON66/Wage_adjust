# 跨 AI 计划评审 — Phase 6

> 注：Claude CLI 在当前 Windows 环境下无法启动独立会话，Gemini CLI 未安装。本次仅使用 Codex (GPT-5.4) 完成评审。

## Codex 评审 (GPT-5.4)

### Plan 01: 后端

**概要**
技术方向正确：逐行 SAVEPOINT、HTTP 207、员工 upsert 和 xlsx 生成覆盖了大部分核心成功标准。风险不在架构，而在缺失的实现细节：upsert 的唯一约束保障、行失败后的会话恢复、认证导入路径的明确化，以及清晰的文件/响应契约。

**优势**
- `db.begin_nested()` 很好地匹配 IMP-01 和 IMP-02 的部分成功需求
- 返回 `207 Multi-Status` 与产品决策和成功标准一致
- employee_no upsert + 审计日志直接解决 IMP-05 和 D-03
- 复用 openpyxl 生成模板/错误报告是务实方案
- 同步处理匹配 D-07，避免不必要的队列复杂度

**关注点**
- `[HIGH]` 除非 `employee_no` 有数据库唯一约束/索引，否则 upsert 不可靠。仅应用层查找不足以防止重复
- `[HIGH]` 认证导入行为不明确。阶段范围包含认证记录，但计划细节以员工为中心
- `[HIGH]` 仅 SAVEPOINT 不够；行级 DB 失败后 SQLAlchemy 会话必须正确清理，否则后续行可能因被污染的会话状态而失败
- `[MEDIUM]` 完全解析后才检查 5000 行限制，浪费了超大文件的 CPU/内存
- `[MEDIUM]` "GBK 编码的 Excel 文件"在技术上模糊。编码对 CSV 重要，对 .xlsx 不适用
- `[MEDIUM]` 文件验证不充分：允许的扩展名、文件大小上限、恶意 xlsx 文件处理未提及
- `[LOW]` 修改 import_job.py 如果暗示持久化任务语义，可能构成范围蔓延

**建议**
- 添加 employee_no 的唯一约束迁移任务
- 明确认证导入规则：键查找、重复处理、预期的行级失败
- 现在定义精确的逐行错误载荷：行号、导入类型、字段/错误消息
- 在完整解析前添加早期检查：文件大小、扩展名/类型
- 澄清文件格式矩阵：.xlsx + CSV (UTF-8 BOM/GB18030/GBK)

**风险评估：MEDIUM**

### Plan 02: 测试

**概要**
意图良好，但"RED 测试桩"对可靠性阶段来说太弱。此阶段需要行为和集成测试来证明部分提交、幂等重导入、真实编码处理和模板正确性。

**优势**
- 将测试工作映射到所有六个导入需求
- Wave 1 早期运行测试是正确的风险消除策略
- 独立测试文件保持关注点隔离

**关注点**
- `[HIGH]` "桩"不够。阶段成功标准需要端到端行为检查，不是占位覆盖
- `[HIGH]` 没有明确提及 GBK/GB18030 CSV 或 xlsx 验证的真实夹具文件
- `[MEDIUM]` 幂等性需要 DB 状态断言，不仅仅是响应断言
- `[MEDIUM]` 尽管在阶段范围内，认证导入测试不明确
- `[MEDIUM]` 如果在响应契约确定前编写测试，可能编码不稳定假设

**建议**
- 至少一个 API 集成测试套件作为门控：混合 100 行导入、207 响应、90 成功/10 失败
- 在仓库中使用真实夹具文件：UTF-8 BOM CSV、GBK/GB18030 CSV
- 直接断言 DB 副作用：提交行数、审计日志创建、无重复员工
- 添加认证场景：缺失员工引用和行级失败

**风险评估：HIGH**

### Plan 03: 前端

**概要**
组件拆分合理，模板下载匹配部分阶段目标。主要缺口是缺少可下载的 Excel 错误报告，以及大量失败行的 UX 不够健壮。

**优势**
- 汇总和错误表格视图拆分为独立组件，清晰可维护
- 依赖后端工作是正确的排序
- 双格式模板下载匹配 IMP-06 和 D-04
- 人工验证检查点适合验证导入 UX

**关注点**
- `[HIGH]` D-02 要求页面表格 + 可下载 Excel 错误报告。计划只提及表格和模板下载
- `[MEDIUM]` 大量失败行可能导致渲染/性能问题
- `[MEDIUM]` 前端对 207 响应的 API 契约未明确提出
- `[MEDIUM]` D-07 要求同步处理 + loading 状态，但加载/超时/错误状态行为不清晰
- `[LOW]` 如果前端独立拥有列头定义，模板下载可能与后端接受的格式漂移

**建议**
- 添加明确的下载导入后错误报告任务
- 定义大结果集 UX：内联预览 + 完整可下载报告
- 验证 API 客户端将 207 视为成功并渲染部分成功响应
- 包含加载/超时处理和同步导入的重试行为

**风险评估：MEDIUM**

## 共识总结

### 核心关注点（按优先级）

1. **[HIGH] 测试计划过于薄弱** — "RED 测试桩"对可靠性阶段不足，需要真实的行为测试和集成测试，包含真实编码夹具文件和 DB 状态断言
2. **[HIGH] 前端缺少可下载错误报告** — D-02 明确要求页面表格 + 可下载 Excel 错误报告，但 Plan 03 未包含后者
3. **[HIGH] 认证导入路径不明确** — 阶段范围包含认证记录，但三个计划均以员工导入为中心，认证导入的键查找、重复处理、测试覆盖均不充分
4. **[HIGH] SAVEPOINT 后会话清理** — 行级失败后 SQLAlchemy 会话状态恢复未明确，可能导致后续行连锁失败
5. **[MEDIUM] employee_no 唯一约束** — upsert 可靠性依赖数据库级唯一约束，仅应用层查找不足

### 已确认的优势

- 三计划结构（后端/测试/前端）分层清晰
- SAVEPOINT + HTTP 207 架构选择正确
- 同步处理避免不必要的复杂度
- 组件拆分（ImportResultPanel + ImportErrorTable）合理
- 双格式模板下载覆盖 IMP-06 和 D-04

### 分歧观点

无（仅有一个评审者）。建议安装 Gemini CLI 后重新运行以获得多视角评审。
