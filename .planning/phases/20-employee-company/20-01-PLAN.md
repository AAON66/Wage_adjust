## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Alembic migration -> live employee schema | company 列若缺失或类型不一致，manual/import 路径会在运行时漂移 |
| uploaded employee table -> ImportService upsert logic | 列存在与否直接决定 company 是否被覆盖、清空或保留 |
| admin / HR write path -> employee profile storage | 未统一 trim/clear 语义会让 manual 和 import 行为不一致 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering | `alembic/versions/*_add_company_to_employee.py` + `backend/app/models/employee.py` | mitigate | 迁移与 ORM 同时更新，并用 grep-verifiable migration acceptance criteria 锁定 `company` 列定义 |
| T-20-02 | Integrity | `backend/app/services/import_service.py` | mitigate | 对 `company` 使用显式 `has_company_column` 分支，避免“缺列即清空”破坏现有数据 |
| T-20-03 | Repudiation | import template + API tests | mitigate | CSV/XLSX/API 三条路径都增加 company 断言，避免只改一条路径后仍声称 Phase 20 完成 |
| T-20-04 | Tampering | `backend/app/services/employee_service.py` | mitigate | manual create/update 在服务层统一 trim / clear，避免不同前端提交格式导致存储值分裂 |
