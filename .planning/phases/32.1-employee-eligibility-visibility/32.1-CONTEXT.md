# Phase 32.1: 员工端调薪资格自助可见 (紧急上线) - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

员工登录后，在自己的 `MyReview` 页面（`/my-review`）随时看到本人当前的调薪资格状态（合格 / 不合格 / 需审核）和未通过规则原因，不依赖 HR 配置活动调薪周期。是 Phase 35「员工端自助体验」完整版的紧急 MVP 切片。

**本期交付（4 项 ESELF 需求）:**
- ESELF-01 资格三态展示
- ESELF-02 未通过规则脱敏说明
- ESELF-04 无参数路由 + 跨员工 403
- ESELF-05 「数据更新于」时间戳

**不在本期（推迟到 Phase 35）:**
- ESELF-03 绩效档次（1/2/3 档）— 依赖 Phase 33 `PerformanceTierEngine` + Phase 34 `/api/v1/performance` 路由，本期不动

</domain>

<decisions>
## Implementation Decisions

### 范围切割 (本期 vs Phase 35)

- **D-01:** Phase 32.1 = ESELF-01 + ESELF-02 + ESELF-04 + ESELF-05；ESELF-03 绩效档次推迟到 Phase 35（依赖 33/34）
- **D-02:** 全量上线，**不引入 feature flag**。MVP 直接对所有员工角色可见，问题靠 hotfix 修复（紧急上线节奏）
- **D-03:** Phase 35 完成时仅需在本期已建好的「调薪资格」卡片下方追加绩效档次徽章 + 完整无参数路由 `/performance/me/tier`，不重构本期代码

### 后端 API

- **D-04:** 新增 `GET /api/v1/eligibility/me`（无参数路由，符合 ESELF-04）
  - 鉴权：`Depends(get_current_user)`，自动取 `current_user.employee_id`
  - 若 `current_user.employee_id` 为空（未绑定员工）→ 返回 422 + 中文提示「您尚未绑定员工信息」
  - 若 `employee_id` 在数据库中找不到 employee → 404
- **D-05:** **不修改**现有 `GET /api/v1/eligibility/{employee_id}`（仍由 `admin/hrbp/manager` 使用）。本期不强制员工角色 403，留给 Phase 35 收口
- **D-06:** Service 层复用 `EligibilityService.check_employee()`（已存在），不新增 Engine 逻辑。本期是「自访问 + 数据脱敏视图」的封装

### 数据脱敏 (ESELF-02)

- **D-07:** **直接复用 `RuleResult.detail` 字段**（已审计：当前 4 条规则的 detail 全是「已入职 X 个月」/「距上次调薪 Y 个月」/「绩效等级为 C」/「假期 X 天」，本身已无 YYYY-MM-DD 或薪资数字，符合 ESELF-02 脱敏要求）
- **D-08:** 不引入 `_sanitize_for_employee` helper，不新增 `employee_safe_detail` 字段。**未来若 detail 中加入敏感数据**（如新规则引入具体日期），由该 phase 自己负责脱敏，本期不预设
- **D-09:** 本期在 `eligibility.py` 添加注释 `# Employee-facing endpoint — RuleResult.detail must remain desensitized`，提醒 future phase 修改时注意

### UI 集成 (ESELF-01)

- **D-10:** 在 `MyReviewPage` 顶部新增独立 section（用 `surface px-6 py-8` 样式，与现有 section 一致），标题「本人调薪资格」
  - 位置：在「当前角色」标头之后，「周期选择 / 上传区」之前
  - 不依赖 cycle 选择 — 进入页面即调用 `/eligibility/me`
- **D-11:** 顶部状态徽章（4 种）：
  - `eligible` → 绿底 `✓ 调薪资格合格`
  - `ineligible` → 红底 `✗ 调薪资格不合格（{N} 项未通过）`
  - `pending` → 黄底 `⏳ 资格待定（{N} 项数据缺失）`
  - 未绑定员工 → 灰底 `请先在 「账号设置」 绑定员工`
- **D-12:** 4 条规则**全部摆开**展示（不折叠），4 色语义对齐：
  - `eligible` 行：绿色左条 + 绿色 status badge `通过`
  - `ineligible` 行：红色左条 + 红色 status badge `未通过`
  - `data_missing` 行：灰色左条 + 灰色 status badge `数据缺失`
  - `overridden` 行：蓝色左条 + 蓝色 status badge `已覆盖`
- **D-13:** 每条规则显示：`rule_label`（粗体）+ `detail`（次级文字 14px text-steel）+ status badge（右侧）
- **D-14:** 顶部右侧显示 `数据更新于 YYYY-MM-DD HH:MM`（small text-steel），格式 `2026-04-22 09:38`（不带秒）

### 时间戳 (ESELF-05)

- **D-15:** `EligibilityResultSchema` 增加 `data_updated_at: datetime` 字段（nullable, ISO 8601）
- **D-16:** Service 层计算 `max(updated_at)`：取 `Employee.updated_at` + 该员工最近一条 `PerformanceRecord.updated_at` + 最近一条 `SalaryAdjustmentRecord.updated_at` + 最近一条 `NonStatutoryLeave.updated_at` 的最大值
  - 若任一 model 没有 `updated_at` 字段 → 跳过该项
  - 若所有数据源都为空 → `data_updated_at = None`，前端显示「数据从未更新」
- **D-17:** 前端用 `Intl.DateTimeFormat('zh-CN', { dateStyle: 'medium', timeStyle: 'short' })` 渲染（避免本地化问题）

### 前端 Service 层

- **D-18:** `frontend/src/services/eligibilityService.ts` 新增 `fetchMyEligibility(): Promise<EligibilityResultWithTimestamp>`，调用 `GET /api/v1/eligibility/me`
- **D-19:** TS 类型 `EligibilityResultWithTimestamp = EligibilityResult & { data_updated_at: string | null }`，存放在 `frontend/src/types/api.ts`
- **D-20:** 错误处理：
  - 422（未绑定）→ 显示「请先绑定员工」灰底卡片 + 跳「账号设置」链接
  - 404（员工不存在）→ 显示「员工档案缺失，请联系 HR」
  - 500/网络错误 → 显示「资格信息暂时不可用，请稍后重试」+ 「重试」按钮

### Claude's Discretion

- 顶部 section 的 padding / 边框圆角具体像素
- 加载状态用 spinner 还是 skeleton
- 重试按钮的具体交互（auto-retry vs 手动）
- API 缓存策略（本期不引入缓存，每次进 page 都查；性能瓶颈未来再说）
- 单元测试的命名风格

</decisions>
