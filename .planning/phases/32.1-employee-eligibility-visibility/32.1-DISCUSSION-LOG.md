## 未通过规则脱敏 + 展示

### Q: ESELF-02 脱敏原则。看了现有 RuleResult.detail，全是「已入职 X 个月」「距上次调薪 Y 个月」「假期 X 天」，本身没有 YYYY-MM-DD 或薪资数字。怎么脱敏？

| Option | Description | Selected |
|--------|-------------|----------|
| 直接复用 detail | 现有 4 条规则的 detail 已经是脱敏后的文本（月份/天数计数，无原始日期/薪资）。Phase 32.1 不动 service，其它 phase 加字段时手动保证。代价最小。 | ✓ |
| 补一个轻量 sanitize 层 | 在 EligibilityResultSchema 增加 employee_safe_detail（可选），默认 = detail。Service 层提供 to_employee_view() 转换。未来加字段不会漏掉。额外 1 人日。 | |
| 双套 detail（重补） | RuleResult 加 internal_detail + employee_detail 两套。最安全但为了「万一」充入推倒重来。现阶段是过度设计。 | |

**User's choice:** 直接复用 detail（推荐）
**Notes:** 现有 detail 已足够脱敏，不引入复杂度。在 eligibility.py 添加注释提醒

### Q: MyReview 中未通过规则的展示形式（判定被拒时）？

| Option | Description | Selected |
|--------|-------------|----------|
| 全部摄开 + 颜色语义 | 4 条规则都列（含通过的），ineligible 红 / pending 黄 / eligible 绿 / data_missing 灰。一眼看到全貌，不需要点击。 | ✓ |

**Notes:** 资格只有 4 条，无需折叠

## MyReview 页面集成位置

### Q: 「调薪资格」卷插哪里？

| Option | Description | Selected |

| 顶部独立区 + 区别卡 | 在周期状态之上加一个独立 section，独立头条「本人调薪资格」。随时可见，不依赖选中 cycle，符合 ESELF-01「不依赖调薪周期」。 | ✓ |

## 「数据更新于」时间戳来源 (ESELF-05)

### Q: 时间戳取什么时间？

| Option | Description | Selected |
|--------|-------------|----------|
| 各数据源 max(updated_at) | 后端返回 max(employee.updated_at, perf.updated_at, salary_adj.updated_at, leave.updated_at)。真实反映「哪个数据最近被改」。 | ✓ |
| 接口调用时刻 | 后端返回 now()。意义是「查询时间」。最简但价值低（总是「刚刚」）。 | |
| 最近一次同步/导入时间 | 用 FeishuSyncLog 或 ImportJob 的最近完成时间。反映「上次同步」但不反映手动修改。 | |

**User's choice:** 各数据源 max(updated_at)（推荐）
**Notes:** 真实反映数据新鲜度
