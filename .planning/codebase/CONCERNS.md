# Codebase Concerns

**Analysis Date:** 2026-03-25

---

## Security Concerns

### `.env` File Committed to Git
- **Priority:** HIGH
- Issue: `.env` is tracked by git (`git ls-files` confirms it is present in the index). The `.gitignore` excludes `.env*` but the file was committed before the rule took effect. Any secrets stored in it (API keys, DB passwords) are permanently in git history.
- Files: `.env`, `.gitignore`
- Current mitigation: None — the file is in version control.
- Recommendations: Rotate all secrets, run `git rm --cached .env`, verify `.gitignore` excludes the file, and audit git history for leaked values.

### Default Placeholder Secrets Accepted at Runtime
- **Priority:** HIGH
- Issue: Config defaults include `jwt_secret_key = "change_me"`, `deepseek_api_key = "your_deepseek_api_key"`, `public_api_key = "your_public_api_key"`, `storage_access_key = "your_access_key"`, and `storage_secret_key = "your_secret_key"`. The application starts and handles traffic normally when none of these are overridden. JWTs signed with `"change_me"` are cryptographically worthless.
- Files: `backend/app/core/config.py` lines 25, 43, 66, 32–33
- Current mitigation: `DeepSeekService._is_configured()` detects the placeholder API key and silently falls back to stub data — but the JWT key and public API key have no equivalent startup guard.
- Recommendations: Add a startup validation step that raises on known placeholder values when `environment != "development"`. At minimum, warn loudly to logs.

### No Brute-Force or Account Lockout Protection on Login
- **Priority:** HIGH
- Issue: `POST /api/v1/auth/login` performs a direct password verify with no rate limiting, failed-attempt counter, CAPTCHA, or lockout. An attacker can enumerate passwords at request-rate speed.
- Files: `backend/app/api/v1/auth.py` lines 78–91
- Current mitigation: None.
- Recommendations: Add slowapi or equivalent per-IP rate limiting on the login endpoint, or track failed attempts per email and implement temporary lockout.

### ID Card Numbers (National ID) Stored in Plaintext
- **Priority:** HIGH
- Issue: Chinese national ID numbers are stored as plain `VARCHAR(32)` in both `employees` and `users` tables. These are high-sensitivity PII under China's Personal Information Protection Law (PIPL). No encryption, hashing, or masking is applied anywhere in the codebase.
- Files: `backend/app/models/employee.py` line 15, `backend/app/models/user.py` line 16, `backend/app/schemas/employee.py` lines 11, 27, `backend/app/schemas/user.py` lines 19, 32
- Current mitigation: None.
- Recommendations: Encrypt at rest using a reversible symmetric key (for lookup) or store only a salted hash. Mask the value in all API responses (e.g., show only last 4 digits).

### Salary Figures Exposed to `employee` Role via Recommendation Endpoints
- **Priority:** HIGH
- Issue: `GET /api/v1/salary/{recommendation_id}` and `GET /api/v1/salary/by-evaluation/{evaluation_id}` use `get_current_user` (no role check), relying only on `ensure_recommendation_access` / `ensure_evaluation_access` for scope gating. Employees can retrieve their own `current_salary` and `recommended_salary` in full precision. Whether this is intentional or a leak depends on HR policy, but there is no access-level differentiation in the schema.
- Files: `backend/app/api/v1/salary.py` lines 71–117
- Current mitigation: AccessScopeService limits records by employee ownership, but does not restrict which fields are returned.
- Recommendations: Add a role-aware response filter: return full salary figures only to `admin`, `hrbp`, `manager`; return adjustment ratio only to `employee`.

### Public API Rate Limit Config is Unused
- **Priority:** MEDIUM
- Issue: `public_api_rate_limit: str = "1000/hour"` is defined in config but never wired to a middleware or dependency. The public API (`/api/v1/public/...`) has no actual rate limiting — only API key authentication.
- Files: `backend/app/core/config.py` line 67, `backend/app/api/v1/public.py`
- Current mitigation: API key required for access.
- Recommendations: Implement rate limiting with slowapi or a Redis-backed counter using the configured limit string.

### Storage Path Traversal Not Validated on Read
- **Priority:** MEDIUM
- Issue: `LocalStorageService.resolve_path(storage_key)` resolves the path but does not assert that the result stays within `base_dir`. A crafted `storage_key` containing `../../` segments could escape the uploads directory.
- Files: `backend/app/core/storage.py` lines 23–35
- Current mitigation: Storage keys are generated by the application at write time (UUID prefixed), so attacker-supplied keys would only appear via direct DB manipulation. However, if a storage key is ever derived from user input the path is unguarded.
- Recommendations: Add `assert resolved_path.is_relative_to(self.base_dir)` before read/delete operations.

### Password Strength Enforced Only in Frontend
- **Priority:** MEDIUM
- Issue: The backend enforces only `min_length=8` for passwords (`backend/app/schemas/user.py`). Password strength scoring exists only in `frontend/src/utils/password.ts` and is advisory only. API clients bypass all strength requirements.
- Files: `backend/app/schemas/user.py` lines 30, 41, 45–46; `frontend/src/utils/password.ts`
- Current mitigation: None on backend.
- Recommendations: Add a backend validator that rejects passwords below a minimum complexity threshold (e.g., requires mixed case + digit or symbol).

### No HTTPS / TLS Enforcement
- **Priority:** MEDIUM
- Issue: Neither uvicorn configuration, CORS policy, nor documentation enforces HTTPS. Tokens, salary data, and ID card numbers would be sent in plaintext over HTTP if the deployment does not independently handle TLS.
- Files: `backend/app/main.py`, `start_backend_local.cmd`, `start_project_local.cmd`
- Current mitigation: None in application layer.
- Recommendations: Document that a TLS-terminating reverse proxy (nginx, Caddy) is required before production deployment.

## Performance Risks

### Dashboard and Public API Load All Submissions Into Memory
- **Priority:** HIGH
- Issue: `DashboardService._submissions()` and `DashboardService._evaluations()` load the entire `EmployeeSubmission` and `AIEvaluation` tables with eager-loaded relationships before applying Python-side access filtering. With thousands of employees and multiple cycles this becomes a full-table scan + N relationship loads on every dashboard request.
- Files: `backend/app/services/dashboard_service.py` lines 30–55
- Impact: Response times grow linearly with dataset size. Memory usage spikes per request.
- Fix approach: Push access-scope filtering into the SQL query via a subquery or JOIN against `user_departments`, and apply cycle/department filtering at the SQL layer before loading.

### Import Service and Several APIs Have No Pagination
- **Priority:** HIGH
- Issue: `ImportService.list_jobs()` returns all import jobs. `SubmissionService` returns all submissions for a cycle. `ApprovalService` returns all approval records. None of these endpoints apply SQL-level `LIMIT`/`OFFSET`.
- Files: `backend/app/services/import_service.py` line 105, `backend/app/services/submission_service.py` line 35, `backend/app/services/approval_service.py` lines 79, 259, 340, 364, 386
- Impact: Lists grow unboundedly as data accumulates.
- Fix approach: Add `page`/`page_size` query parameters like the `employees` endpoint already does, and push `LIMIT`/`OFFSET` into the SQLAlchemy query.

### LLM Calls Execute Synchronously Inside FastAPI Route Handlers
- **Priority:** HIGH
- Issue: `EvaluationService.generate_evaluation()` and `ParseService.parse_submission_files()` call `DeepSeekService._invoke_json()` which uses synchronous `httpx.Client()`. These calls block the Uvicorn worker thread for up to 120 seconds (`deepseek_evaluation_timeout_seconds`). While FastAPI is async, all these routes are defined as regular `def` (not `async def`), so FastAPI runs them in a thread pool — but a long LLM call still occupies one thread for the full duration, limiting throughput under concurrent load.
- Files: `backend/app/services/llm_service.py` lines 275–278, `backend/app/api/v1/evaluations.py` lines 69–114
- Impact: A single batch of evaluations can exhaust the Uvicorn thread pool, making other API calls unresponsive.
- Fix approach: Move LLM calls to a Celery/background task queue (already in `requirements.txt` but unused). Return a `202 Accepted` with a task ID, and poll for completion.

### Files Are Read Entirely Into Memory Before Validation
- **Priority:** MEDIUM
- Issue: `upload.file.read()` loads the complete file into a `bytes` object before size validation. A 200 MB upload is fully buffered in memory before the application checks whether it exceeds the limit.
- Files: `backend/app/services/file_service.py` lines 119–139, `backend/app/services/employee_handbook_service.py` line 71, `backend/app/services/import_service.py` line 134
- Impact: Memory spikes proportional to upload size; DoS risk if many concurrent uploads arrive.
- Fix approach: Check `Content-Length` header early, or read in chunks and abort when the running total exceeds the limit.

### ROI Calculation in Dashboard Uses Arbitrary Formula
- **Priority:** MEDIUM
- Issue: `DashboardService._estimate_roi_multiple()` computes ROI using an invented formula (`impact_factor * multiplier_factor / adjustment_factor`) with no business basis documented. Results appear in public-facing dashboard and potentially in the public API.
- Files: `backend/app/services/dashboard_service.py` lines 93–98
- Impact: Misleading KPI data presented to HR and executives.
- Fix approach: Replace with a formula agreed upon by stakeholders, or suppress the ROI field until it is properly defined.

## Reliability Risks

### Schema Migrations Bypass Alembic with Raw `ALTER TABLE` on Startup
- **Priority:** HIGH
- Issue: `ensure_schema_compatibility()` runs raw `ALTER TABLE ADD COLUMN` SQL on every application startup, acting as a hand-written migration system parallel to Alembic. Four Alembic migration files exist but are not executed automatically. The startup migration only handles SQLite-compatible `ADD COLUMN` and cannot handle renames, drops, index changes, or PostgreSQL-specific syntax.
- Files: `backend/app/core/database.py` lines 73–115, `alembic/versions/`
- Impact: Schema drift between SQLite (dev) and PostgreSQL (prod); migration failures in production when more complex changes are needed.
- Fix approach: Retire `ensure_schema_compatibility()` entirely and use `alembic upgrade head` as the sole migration path. Add a deployment step that runs migrations before starting the server.

### Celery, Redis, and MinIO/S3 Are in `requirements.txt` But Not Used
- **Priority:** HIGH
- Issue: `celery`, `redis`, `hiredis`, `minio`, `boto3`, `asyncpg`, `aiohttp` are installed dependencies with no usage in `backend/app/`. No Celery app is defined, no Redis client is instantiated in application code, no MinIO/S3 client is used. The `redis_url` config key exists but is never read. File storage is `LocalStorageService` (local filesystem only).
- Files: `requirements.txt`, `backend/app/core/config.py` line 64, `backend/app/core/storage.py`
- Impact: False impression that async tasks and object storage are implemented; production deployments assume local disk for all uploaded files, which is lost on container restart.
- Fix approach: Either implement Celery task queues and cloud storage as intended, or remove unused dependencies and document that they are planned.

### OCR for Image Evidence Is Not Implemented
- **Priority:** HIGH
- Issue: `ImageParser.parse()` returns a stub string: `"OCR is reserved for a later task."` Images (`.png`, `.jpg`, `.jpeg`) are accepted and stored, but no text is extracted. Any employee who uploads image-based evidence (screenshots, certificates, award letters) receives zero meaningful evidence content fed to the LLM evaluator.
- Files: `backend/app/parsers/image_parser.py` lines 13–22
- Impact: Silent evaluation degradation for image-heavy submissions. The user sees no error, but AI scoring ignores all image content.
- Fix approach: Integrate an OCR library (pytesseract, easyocr, or a cloud vision API) into `ImageParser.parse()`.

### LLM Fallback Silently Produces Stub Evaluations
- **Priority:** HIGH
- Issue: When DeepSeek is unconfigured or fails, `DeepSeekService._invoke_json()` returns `fallback_payload` — a caller-supplied stub. The evaluation is stored as `generated` status with placeholder scores. There is no flag in the evaluation model to distinguish "LLM-backed" from "stub" evaluations. HR reviewers could approve a stub salary recommendation without realising the AI never ran.
- Files: `backend/app/services/llm_service.py` lines 218, 252, `backend/app/services/evidence_service.py` line 163
- Impact: Unreliable evaluation data in production if keys are misconfigured.
- Fix approach: Add an `llm_used: bool` field to `AIEvaluation`. Prevent `recommend_salary` from proceeding when `llm_used=False` unless explicitly overridden by an admin.

### Single Point of Failure: No Database Connection Pooling for SQLite
- **Priority:** MEDIUM
- Issue: The app defaults to SQLite. SQLite with `check_same_thread=False` allows multiple threads but serialises all writes. Under concurrent requests (uploads, evaluations, approvals) write operations will block each other. The configured `database_pool_size=10` and `database_max_overflow=20` only apply to non-SQLite engines.
- Files: `backend/app/core/database.py` lines 26–38
- Impact: Write throughput collapses under modest concurrency in the default configuration.
- Fix approach: Document that PostgreSQL is required for any multi-user production load, and add an environment check that refuses to start with SQLite when `environment == "production"`.

### No File Cleanup on Failed Uploads or Evaluations
- **Priority:** MEDIUM
- Issue: `FileService._create_file_record()` writes bytes to disk, then adds the DB record. If `db.commit()` fails after `storage.save_bytes()`, the file is orphaned on disk. Similarly, failed parse runs leave `parse_status='failed'` files whose bytes are never cleaned up.
- Files: `backend/app/services/file_service.py` lines 98–113, 119–139
- Impact: Disk accumulates orphaned files over time.
- Fix approach: Wrap storage write + DB commit in a try/except that calls `storage.delete()` on any DB error.

## Technical Debt

### Dual Schema-Migration Systems
- **Priority:** HIGH
- Issue: The project has both Alembic (`alembic/versions/`) and a hand-rolled `ensure_schema_compatibility()` in `database.py`. They apply overlapping changes to different tables. As the schema grows this divergence will cause conflicts.
- Files: `backend/app/core/database.py` lines 73–115, `alembic/versions/`
- Fix approach: Consolidate to Alembic only (see Reliability section above).

### Salary Engine Rules Are Hardcoded Module-Level Constants
- **Priority:** HIGH
- Issue: `LEVEL_RULES`, `JOB_LEVEL_ADJUSTMENTS`, `DEPARTMENT_ADJUSTMENTS`, and `JOB_FAMILY_ADJUSTMENTS` are Python module-level dicts in `backend/app/engines/salary_engine.py`. The CLAUDE.md requirement states all coefficients must be configurable, not hardcoded. Changing any salary multiplier or floor/ceiling requires a code change and redeployment.
- Files: `backend/app/engines/salary_engine.py` lines 7–44
- Fix approach: Move these to a database-backed config table or a YAML/JSON file loaded at startup, accessible and editable by admins through the `/system` API.

### `EvaluationDetail.tsx` and `SalarySimulator.tsx` Are Oversized
- **Priority:** MEDIUM
- Issue: `EvaluationDetail.tsx` is 2077 lines and `SalarySimulator.tsx` is 1467 lines. These pages embed component logic, data fetching, state management, and rendering in a single file. This makes them hard to test, extend, or debug.
- Files: `frontend/src/pages/EvaluationDetail.tsx`, `frontend/src/pages/SalarySimulator.tsx`
- Fix approach: Extract sub-components and custom hooks; each file should stay under ~400 lines.

### Dashboard Charts Are Hand-Drawn SVG
- **Priority:** MEDIUM
- Issue: `SalaryHistoryPanel.tsx` renders charts using raw SVG path calculations with hardcoded coordinate math. `DistributionChart.tsx` and `HeatmapChart.tsx` follow similar patterns. This is fragile, hard to maintain, and provides no accessibility (no ARIA, no keyboard navigation).
- Files: `frontend/src/components/salary/SalaryHistoryPanel.tsx`, `frontend/src/components/dashboard/DistributionChart.tsx`, `frontend/src/components/dashboard/HeatmapChart.tsx`
- Fix approach: Introduce a proper charting library (recharts, visx, or Chart.js). The frontend has no chart dependency currently.

### `psycopg2-binary` and `asyncpg` Installed but Unused
- **Priority:** LOW
- Issue: Both PostgreSQL drivers are in `requirements.txt` but the application only connects to SQLite in the current codebase. `asyncpg` is the async driver, but all SQLAlchemy sessions use the synchronous `Session` class.
- Files: `requirements.txt` lines 4–5
- Fix approach: Remove `asyncpg` and `aiohttp` unless async SQLAlchemy sessions are actively planned. Keep `psycopg2-binary` only.

## Configuration Gaps

### No `.env.example` File Is Enforced as a Checklist
- **Priority:** MEDIUM
- Issue: `.env.example` exists but does not document which variables are mandatory for production vs optional. New deployers may not realise `JWT_SECRET_KEY`, `DEEPSEEK_API_KEY`, and `PUBLIC_API_KEY` must be replaced before going live.
- Files: `.env.example`
- Fix approach: Add inline comments to `.env.example` marking `REQUIRED` fields; add startup validation as noted in the Security section.

### CORS Allows All `localhost` Origins by Default
- **Priority:** MEDIUM
- Issue: The default `backend_cors_origins` list contains five localhost/127.0.0.1 variants. If `BACKEND_CORS_ORIGINS` is not set in production, the API will refuse all requests from the deployed frontend origin while allowing dev-machine origins.
- Files: `backend/app/core/config.py` lines 55–63
- Fix approach: Default to an empty list in non-development environments, and require explicit configuration of allowed origins.

### `public_api_key` Has a Placeholder Default
- **Priority:** MEDIUM
- Issue: The default value `"your_public_api_key"` is weak and predictable. The public API exposes `current_salary` and `recommended_salary` for all employees in a cycle to any caller who knows this key.
- Files: `backend/app/core/config.py` line 66
- Fix approach: Validate at startup that `public_api_key` is not the placeholder value (same recommendation as JWT key).

## Missing Critical Features

### No Async Job Queue for LLM and Parse Operations
- **Priority:** HIGH
- Issue: File parsing and evaluation generation are long-running operations (up to 120 seconds) that execute synchronously during HTTP requests. Celery and Redis are listed as dependencies but no worker or task is defined anywhere in the codebase. Users must hold the connection open for the full duration.
- Blocks: Reliable batch evaluation, scalable parse-all workflows, background retry on LLM failure.

### No OCR for Image Files
- **Priority:** HIGH
- Issue: See Reliability section. Image evidence produces no extractable text. This is a core feature of the stated design (support for PNG evidence materials) that is explicitly deferred with a stub comment.
- Files: `backend/app/parsers/image_parser.py`
- Blocks: Evaluation quality for any image-based submission.

### No Frontend Tests
- **Priority:** HIGH
- Issue: There are zero test files in `frontend/src/`. No test runner (vitest, jest) is configured. No `@testing-library/react` is installed. Critical frontend paths (login, file upload, evaluation display, salary simulation) have no automated coverage.
- Files: `frontend/package.json`
- Risk: Any frontend regression is invisible until manual QA.

### Public API Has No Per-Client Key Rotation or Revocation
- **Priority:** MEDIUM
- Issue: There is one global `public_api_key` shared across all external integrations. If the key is leaked, all integrations must be updated simultaneously. There is no key management interface.
- Files: `backend/app/api/v1/public.py`, `backend/app/core/config.py` line 66
- Fix approach: Store API keys in a database table with per-key metadata (owner, created_at, last_used_at) and a revocation flag.

### No Email Notification System
- **Priority:** MEDIUM
- Issue: The workflow requires HR reviewers, managers, and approvers to take action at various stages. There is no email or push notification when a submission needs review, when an approval decision is made, or when a salary recommendation is locked. Actors must manually check the platform.
- Blocks: Practical usability in a real enterprise deployment.

## Scalability Limitations

### SQLite Is the Only Tested Database
- **Priority:** HIGH
- Issue: All tests use an in-memory or temp-file SQLite database. PostgreSQL-specific SQL (e.g., full-text search, JSONB, concurrent writes) is untested. The `ensure_schema_compatibility()` migration path is SQLite-only.
- Fix approach: Add a CI configuration that runs the test suite against PostgreSQL.

### In-Memory Rate Limiter Is Per-Process
- **Priority:** MEDIUM
- Issue: `InMemoryRateLimiter` in `llm_service.py` tracks timestamps in a Python `deque` scoped to the current process. With multiple Uvicorn workers (or Gunicorn workers), each process maintains its own rate-limit window — a caller effectively gets `N × limit` requests per minute across `N` workers.
- Files: `backend/app/services/llm_service.py` lines 29–41
- Fix approach: Use a Redis-backed atomic counter for rate limiting (Redis is already in `requirements.txt`).

### Dashboard Aggregation Has No Caching
- **Priority:** MEDIUM
- Issue: Every `/api/v1/dashboard/*` request recomputes all aggregations from raw submission/evaluation rows with multiple eager-loaded relationships. There is no caching layer (Redis is unused). Under a large dataset a single dashboard page load triggers many full-table scans.
- Files: `backend/app/services/dashboard_service.py`
- Fix approach: Cache computed dashboard results in Redis with a short TTL (e.g., 60 seconds), or pre-aggregate via a scheduled Celery task.

## Dependency Risks

### `python-jose` Has Known Vulnerabilities
- **Priority:** HIGH
- Issue: `python-jose[cryptography]==3.3.0` is used for JWT encoding/decoding. This library has had CVEs related to algorithm confusion attacks (CVE-2024-33664, CVE-2024-33663). It is largely unmaintained.
- Files: `requirements.txt` line 8, `backend/app/core/security.py`
- Risk: JWT tokens could be forged if the algorithm confusion path is exploitable in this configuration.
- Migration plan: Migrate to `PyJWT` (actively maintained, simpler API). The `decode_token` and `_build_token` functions in `security.py` are the only callers; the change is localised.

### `passlib[bcrypt]` Uses a Deprecated Compatibility Layer
- **Priority:** MEDIUM
- Issue: `passlib==1.7.4` is pinned to a version from 2020. The library has known warnings about bcrypt version incompatibilities and the project is in maintenance-only mode. The `pbkdf2_sha256` scheme is used here (not bcrypt), but the package itself is stale.
- Files: `requirements.txt` line 9, `backend/app/core/security.py` line 11
- Migration plan: Switch to `argon2-cffi` or use `bcrypt` directly with `PyJWT`.

### Content-Length Bypass on Large Uploads
- **Priority:** MEDIUM
- Issue: File uploads are read fully into memory with `upload.file.read()` before size checking. FastAPI/Starlette does not enforce `Content-Length` limits at the framework layer by default. Without a reverse proxy that enforces a body size limit, a malicious client can upload files larger than `max_upload_size_mb` by simply not sending a `Content-Length` header.
- Files: `backend/app/services/file_service.py` lines 119–127
- Fix approach: Add `LimitUploadSize` middleware (available in Starlette) with a hard byte cap matching `max_upload_size_mb`.

## Test Coverage Gaps

### Frontend: Zero Coverage
- **Priority:** HIGH
- What is not tested: All UI interactions, form validation, API error handling, token refresh logic, role-based rendering, salary simulation UI.
- Files: `frontend/src/` (no `.test.*` files exist)
- Risk: Regressions in critical user-facing flows are invisible.

### Backend Parsers: Only `code_parser` Is Tested
- **Priority:** MEDIUM
- What is not tested: `PPTParser`, `ImageParser`, `DocumentParser`, archive extraction logic, corrupted file handling.
- Files: `backend/tests/test_parsers/test_code_parser.py` (only file present)
- Risk: Silent parsing failures for non-code evidence types.

### No End-to-End Test for the Full Evaluation Flow
- **Priority:** MEDIUM
- What is not tested: The chain from file upload → parse → evidence extraction → LLM evaluation → salary recommendation → approval → lock is not tested as an integrated sequence. Individual service tests exist, but the cross-service state machine has no coverage.
- Risk: Integration bugs between services (e.g., stale evaluation status after re-generate, orphaned approval records) go undetected.

### No Test for `ensure_schema_compatibility`
- **Priority:** MEDIUM
- What is not tested: The startup migration logic that applies raw SQL is not covered by any test. A regression here would silently corrupt schema on startup.
- Files: `backend/app/core/database.py` lines 73–115, `backend/tests/test_core/test_database.py`
- Risk: Schema migration failures on deployment.
